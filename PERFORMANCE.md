# Shinkansen 效能報告

> 測試版本：v1.0.23 ｜ 測試日期：2026-04-11

## 結論

Shinkansen 翻譯整頁維基百科（一千段文字），瀏覽器記憶體不增反減（中文比英文精簡），網頁不會變卡、不會變慢、翻完之後也不會留下任何多餘的東西。

## 白話文摘要

我們拿了一個非常長的英文維基百科頁面（Taiwan 條目，1,211 段文字、超過 22 萬個字元）來做壓力測試，結果如下：

**記憶體不增反減。** 翻譯前瀏覽器使用 40.7 MB，翻譯後反而降到 35.3 MB——因為中文譯文比英文原文精簡，DOM 節點數也跟著減少。

**網頁不會變卡。** Extension 超過 95% 的時間都在等雲端 API 回傳翻譯結果，你的電腦幾乎不用做任何額外運算。翻譯過程中可以正常捲動、點擊、操作頁面。

**翻完不留痕跡。** 翻譯完成後，網頁上沒有被塞進多餘的 HTML 元素。DOM 節點數甚至比翻譯前還少（因為中文比英文精簡），頁面結構維持乾淨。

**花費極低。** 用最便宜的模型翻譯這整個千段大頁面，API 費用不到 0.08 美元（約台幣 2.5 元）。翻譯過的內容會自動快取，下次再開同一頁不用重新花錢。

以下是完整的技術測試數據。

## 記憶體佔用（JS Heap）

| 時間點 | JS Heap Used |
|--------|-------------|
| 翻譯前（頁面載入 + content script 注入後） | 40.72 MB |
| 翻譯完成後 | 35.34 MB |
| **增量** | **−5.38 MB** |

翻譯 1,000 段文字後，JS heap 反而下降約 5 MB。原因是中文譯文比英文原文精簡，DOM 節點合併簡化後釋放了記憶體。主要的記憶體資料結構包括：

- **段落快取**（`STATE.cache`，Map）：儲存原文 → 譯文的對照，供重複段落直接命中。
- **原文備份**（`STATE.originalHTML`，Map）：保存每個被替換元素的原始 HTML，供「還原原文」功能使用。
- **Content Guard 快取**（`STATE.translatedHTML`，Map）：防止 SPA 框架覆寫譯文。
- **Log buffer**（Service Worker 端，上限 1,000 條）。

翻譯完成後，頁面上沒有殘留任何 Shinkansen 的 DOM 元素（零個帶 `shinkansen` class 的節點、Toast 已自動消失）。Extension 不會在頁面上留下持久性的 DOM 痕跡。

## DOM 影響

| 指標 | 翻譯前 | 翻譯後 | 變化 |
|------|--------|--------|------|
| DOM 節點數 | 16,286 | 13,109 | **−3,177** |
| 注入的 style 元素 | 0 | 0 | — |
| Shinkansen class 元素 | 0 | 0 | — |

DOM 節點數反而減少，原因是中文譯文較短，部分 inline 元素在覆蓋過程中被合併簡化。Extension 採用「原地替換文字節點」策略，不額外包裝 `<span>` 或 `<div>`，因此不會膨脹 DOM。

### API 延遲分佈

| 指標 | 數值 |
|------|------|
| 平均回應時間 | 5,125 ms |
| 最快回應 | 1,263 ms |
| 最慢回應 | 10,448 ms |

翻譯期間超過 95% 的時間花在等待 API 回應。Content script 與 Service Worker 在等待期間幾乎是閒置的，不佔用 CPU 資源。

本地端 CPU 佔用集中在兩個短暫時間點：

1. **翻譯啟動時**：`collectParagraphs()` 遍歷整個 DOM tree 進行段落偵測，屬一次性同步操作。
2. **每批翻譯回來時**：DOM 注入（替換文字節點），每次耗時極短。

## 備註

- 本測試在所有快取清空的狀態下進行（worst case），實際使用時快取命中率會顯著降低 API 呼叫次數與總耗時。
- Wikipedia Taiwan 是刻意選擇的壓力測試頁面（1,211 段），一般網頁的段落數遠低於此。
- 翻譯速度主要受 API 延遲與並行批次數限制，與本地效能無關。提高 `maxConcurrent` 可縮短總耗時，但會更快消耗 rate limit 配額。
- 記憶體數據使用 `performance.memory` API（Chrome 限定）量測 JS heap，不含 V8 外部記憶體（如 DOM 本身的 C++ 物件），但差異通常很小。
